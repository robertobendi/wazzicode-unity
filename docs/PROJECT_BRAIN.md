# Project Brain

The "project brain" is a set of files Unity Vibe OS maintains in your Unity project so future Claude sessions don't have to rediscover the project from scratch. Generated by `uvibe brain` (or the `unity_generate_project_brain` MCP tool).

## Files

Lives under `.unity-vibe/`:

| File | Purpose | Auto-generated? |
|---|---|---|
| `project_brain.md` | Human-readable summary of identity / engine / packages / assets / architecture | yes (overwritten each run) |
| `project_brain.json` | Same data, structured | yes |
| `claude_context.md` | Concise, AI-friendly priming context | yes |
| `conventions.md` | Project-specific rules; **never overwritten** if you've edited it | first-run only |
| `config.json` | Safety + runtime config | first-run only |

Planned (later milestones):

| File | Purpose |
|---|---|
| `asset_index.sqlite` | Indexed asset metadata for fast queries |
| `prefab_index.json`, `script_graph.json`, `asset_graph.json` | Graph data |
| `scene_snapshots/`, `runtime_logs/`, `snapshots/`, `reports/`, `tasks/`, `trash/` | Workflow artifacts |
| `known_issues.md`, `feature_history.md` | Hand-edited |
| `action_log.jsonl` | Append-only log of write actions |

## Detection (filesystem-only — no Unity required)

`@uvibe/project-brain` reads:

- `ProjectSettings/ProjectVersion.txt` → Unity version + revision.
- `ProjectSettings/ProjectSettings.asset` → product/company name, bundle id, scripting backend, build target.
- `Packages/manifest.json` → packages, render pipeline (HDRP/URP/Built-in), input system.
- Walks `Assets/` and `Packages/` (skipping `Library`, `Temp`, `obj`, `node_modules`, `.git`, `.unity-vibe`, etc.) to count scenes / prefabs / scripts / `*.asset` / materials / textures / audio.
- Lightweight script heuristics (sample of 400 scripts):
  - MonoBehaviour count.
  - ScriptableObject types (regex `class \w+ : ScriptableObject`).
  - Manager/singleton patterns (`*Manager.cs`, `Singleton<...>`, `static \w+ Instance`).
  - Object pooling, save-system, audio-manager presence.

The Unity bridge is **not** required for brain generation; this means `uvibe brain` works in CI or before opening Unity.

## When to refresh

- After major refactors.
- After adding/removing packages.
- Before starting a new feature so Claude has up-to-date context.
- Automatically as part of `uvibe loop` (planned).

## Reading the brain in tools

`@uvibe/project-brain.readBrain(projectPath)` returns the parsed `Brain` object. `brainAgeMs` returns how stale it is.
